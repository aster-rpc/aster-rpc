Priority 0: fix correctness / safety first
1. Eliminate raw-pointer resurrection in async tasks

This is the biggest issue.

You currently do this pattern in many places:

convert *mut T to usize with ptr_id
spawn an async task
later recover &T with ptr_ref

That means Java could free the handle while the operation is still running, and the spawned task would dereference freed memory. This affects things like:

iroh_ffi_node_close
iroh_ffi_net_close
iroh_ffi_net_closed
iroh_ffi_net_connect
iroh_ffi_net_connect_node_addr
iroh_ffi_net_accept
iroh_ffi_connection_open_bi
iroh_ffi_connection_accept_bi
iroh_ffi_connection_open_uni
iroh_ffi_connection_accept_uni
iroh_ffi_connection_read_datagram
iroh_ffi_connection_closed
iroh_ffi_send_stream_write_all
iroh_ffi_send_stream_finish
iroh_ffi_send_stream_stopped
iroh_ffi_recv_stream_read
iroh_ffi_recv_stream_read_exact
iroh_ffi_recv_stream_read_to_end
blob/doc/gossip async functions in the same pattern

What to do instead:

make each handle internally hold an Arc<Inner>
clone the Arc before spawning
async task owns the clone
free function only drops one Arc

That single refactor will make the whole ABI much safer.

2. Document or enforce “no free while op pending”

Even if you keep the current model briefly, you need a hard rule:

A handle must not be freed while any operation that references it is pending.

For Java, undocumented assumptions like this usually become intermittent crashes.

Better is to remove the assumption entirely with internal refcounting.

3. Make tokio::spawn runtime guarantees explicit

spawn_op assumes a Tokio runtime already exists. If any exported function is called before runtime init, Java will get ugly failures.

Add one of:

an explicit iroh_ffi_runtime_init()
or lazy singleton runtime creation inside the bridge
or guarantee that aster_transport_core already manages this and document it clearly

Right now that contract is not visible at the ABI level.

Priority 1: make it fit Java FFM better
4. Add nonblocking operation probing

iroh_ffi_operation_wait is usable, but not ideal for Java because blocking foreign calls are awkward and pin virtual threads.

Add:

iroh_ffi_operation_is_ready(op, out_ready)
iroh_ffi_operation_wait_timeout(op, timeout_ms)
optionally iroh_ffi_operation_status(op, out_status)

Then the Java layer can poll from an executor thread and complete CompletableFutures cleanly.

5. Add operation cancellation

Right now once an op is spawned, Java can only wait and then take the result.

Add:

iroh_ffi_operation_cancel(op)

Even if underlying cancellation is cooperative, Java needs a cancellation story.

6. Add a typed operation-kind query

You currently rely on the caller knowing which take_* function matches the operation.

Add:

iroh_ffi_operation_kind(op, out_kind)

That makes the Java binding much safer and easier to debug.

7. Consider a central completion queue later

Not mandatory for v1, but a queue-based model would fit Java async wrappers better than one-condvar-per-op.

For now, your current model is acceptable if you add nonblocking readiness checks.

Priority 2: complete the endpoint config surface
8. Add real custom relay support

This is a direct requirement miss.

iroh_ffi_endpoint_config_t supports only:

none
default
disabled
staging

That is not enough if Java must specify custom relays.

Add something like:

typedef struct {
    i32 relay_mode;
    const iroh_ffi_bytes_t* alpns;
    usize alpns_len;
    iroh_ffi_bytes_t secret_key;

    const iroh_ffi_string_t* relay_urls;
    usize relay_urls_len;
} iroh_ffi_endpoint_config_t;

Then map:

relay_mode = CUSTOM
use relay_urls
9. Add secret-key export if stable identity matters

You already support secret-key input. Good.

But Java usually also needs:

export current secret key
maybe export/import endpoint ticket or node address helpers

Add:

iroh_ffi_net_export_secret_key
or equivalent on node/endpoint handle
10. Clarify ALPN ownership and semantics

You support multiple ALPNs in config and single ALPN in some create/connect calls.

Document:

whether create_with_config registers multiple listeners
whether connect must use one of configured ALPNs
whether ALPN bytes are arbitrary binary or expected UTF-8-like values

Java bindings need that nailed down.

Priority 3: tighten memory ownership rules
11. Write explicit ownership rules for every returned type

You have free functions, which is good, but the contract should be written down per type.

Document clearly:

iroh_ffi_string_t: allocated by Rust, caller frees with iroh_ffi_string_free
iroh_ffi_bytes_t: allocated by Rust, caller frees with iroh_ffi_bytes_free
iroh_ffi_node_addr_t: caller frees with iroh_ffi_node_addr_free
returned handle pointers: caller frees with the matching handle free
operation handles: caller frees with iroh_ffi_operation_free after taking result

Without this, Java code will leak or double free.

12. Zero or poison results after take_*

If take_result consumes the stored result, repeated take_* calls return pending/error-ish behavior. That is okay, but document it.

Even better:

define “take exactly once”
add IROH_FFI_ERR_ALREADY_TAKEN
13. Add nullable-result conventions to the header/spec

You have OptionalBytes and OptionalU64. Good.

But Java needs crisp rules:

how “none” is represented
whether empty bytes differs from absent bytes
whether absent u64 uses a flag or a sentinel

Keep the current explicit-present flag style where possible.

14. Audit bool in #[repr(C)] structs

bool often works fine with Java FFM if mapped carefully, but it is not my favorite ABI field type.

For maximum portability across languages, I would consider:

u8 code_present
u8 data_present

instead of Rust bool.

Not mandatory for Java alone, but better as a cross-language ABI.

Priority 4: error model improvements
15. Keep last_error_message, but add stable error codes

Right now most failure detail is only the string in thread-local LAST_ERROR.

Add a more structured error surface:

library-level numeric codes
maybe domain/subcode for transport/docs/blobs/gossip

Java can then map:

timeout
invalid argument
not found
connection closed
relay/discovery failure

without parsing strings.

16. Make error retrieval thread expectations explicit

Because LAST_ERROR is thread-local, Java must read it on the same native-calling thread that observed the failing return code.

That can surprise FFM users if they hop executors.

Document it very explicitly.

Priority 5: API ergonomics / completeness
17. Add timeout-capable APIs for long waits

Functions like:

net_accept
connection_accept_bi
connection_accept_uni
connection_read_datagram
recv_stream_read_to_end
gossip receive/subscription ops

may wait a long time.

At minimum, Java should be able to cancel the operation handle. Better still, some APIs should offer timeout inputs.

18. Add explicit half-close / stream state docs

You already expose:

write_all
finish
stopped
stop
closed

That is good, but Java wrapper authors need exact semantics:

when is finish enough
when does peer observe EOF
difference between finish and close
when stopped completes
how reset/stop code flows
19. Add vectored or chunked write/read later

Not required now, but for throughput you may eventually want:

write from caller-provided direct buffer
avoid always cloning into Vec<u8>

For Java v1 this is okay as-is, but it is a future performance item.

20. Add docs/gossip/blobs result typing consistency

The native surface will be easier to bind if all async service APIs follow the same pattern:

submit
wait/probe
take typed result
free returned resource

You are mostly there already; just keep it very consistent as you expand docs/gossip/blobs.

Function-specific notes
iroh_ffi_endpoint_create_with_config

Good start, but expand config to support custom relay URLs and potentially secret-key export symmetry.

iroh_ffi_node_*_client

These are fine ergonomically, but become safe only once node handle lifetime is safe across async ops.

iroh_ffi_net_connect / iroh_ffi_net_accept

These are exactly the kind of functions that need the handle-lifetime fix first.

iroh_ffi_connection_open_bi / accept_bi / open_uni / accept_uni

Good ABI shape. Keep.

iroh_ffi_send_stream_write_all

Usable for Java, but copies input bytes. Fine for v1.

iroh_ffi_recv_stream_read / read_exact / read_to_end

Reasonable. Document absent-vs-empty semantics clearly.

iroh_ffi_operation_wait

Useful as a primitive, but add readiness probe and timeout variant.

iroh_ffi_last_error_message

Fine, but only if same-thread requirement is documented.

My recommended order of work
Replace raw pointer resurrection with Arc-backed handle internals.
Add operation_is_ready and operation_wait_timeout.
Add custom relay URLs to endpoint config.
Write explicit ownership and threading rules in the header/spec.
Add operation cancellation.
Add stable error code taxonomy.
Optionally add a central completion queue later.
Bottom line

The ABI is now structurally sound enough for Java FFM, but I would not bind it yet until you fix the in-flight handle lifetime issue and add custom relay config.

Those are the two blockers.

After that, the Java layer can quite reasonably wrap this into:

CompletableFuture
Flow.Publisher
Kotlin suspend / Flow

without needing to redesign the native side.
