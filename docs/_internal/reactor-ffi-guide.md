# Reactor FFI Guide

**Status:** Authoritative
**Last updated:** 2026-04-12
**Audience:** Implementers writing a new language binding (Java, .NET, Go, Swift, ...).

This document is the contract between the Rust core and any language binding that wants to receive RPC calls. If you're adding support for a new language, read this end-to-end before writing code.

For the broader C ABI (runtime, nodes, blobs, docs, gossip, hooks), see [FFI_PLAN.md](../../ffi_spec/FFI_PLAN.md). The reactor described here is a layer **on top of** that ABI, dedicated to RPC call dispatch.

---

## What the reactor does

The reactor owns the server-side accept loop in Rust. It accepts iroh QUIC connections, reads stream headers and request payloads, and delivers fully-formed call descriptors to the binding via a lock-free SPSC ring buffer. The binding's poll thread drains call descriptors, dispatches them to user-defined handlers, and submits responses back through the same handle.

```
                  ┌──────────── Rust ─────────────┐    ┌──── Language binding ────┐
QUIC accept ─────►│ accept_loop                   │    │                          │
                  │   ↓                           │    │                          │
                  │ connection task per peer      │    │                          │
                  │   ↓                           │    │                          │
                  │ frame reader (header + req)   │    │                          │
                  │   ↓                           │    │                          │
                  │ tokio mpsc                    │    │                          │
                  │   ↓                           │    │                          │
                  │ pump task (single writer) ────┼───►│ aster_reactor_poll       │
                  │                               │    │   ↓                      │
                  │ SPSC ring (cache-aligned)     │    │ for each call:           │
                  │                               │    │   decode header          │
                  │                               │    │   dispatch to handler    │
                  │                               │    │   build response         │
                  │ ◄─── aster_reactor_submit ────┼────│   submit                 │
                  │                               │    │   release buffers        │
                  │ writes response to QUIC       │    │                          │
                  └───────────────────────────────┘    └──────────────────────────┘
```

The pump task is the only writer to the ring; the binding's poll thread is the only reader. This is a true SPSC pattern with cache-line-padded head/tail and Acquire/Release fencing — no locks, no CAS on the hot path. See `core/src/ring.rs` for the implementation.

## The 5 C functions

All function signatures are in `ffi/iroh_ffi.h` (regenerated by cbindgen). The Rust source is `ffi/src/reactor.rs`. Here are the semantics:

### `aster_reactor_create`

```c
int32_t aster_reactor_create(
    iroh_runtime_t runtime,
    iroh_node_t node,
    uint32_t ring_capacity,
    aster_reactor_t *out_reactor);
```

Attaches a reactor to an existing node and starts its accept loop. `ring_capacity` is rounded up to the next power of two; pass `0` for the default (256 slots).

Returns `IROH_STATUS_OK` on success and writes the reactor handle to `out_reactor`. Returns `IROH_STATUS_NOT_FOUND` if the node handle is invalid, or `IROH_STATUS_INVALID_ARGUMENT` if `out_reactor` is null.

The reactor begins delivering calls immediately. You can call `aster_reactor_poll` from the moment this function returns.

### `aster_reactor_poll`

```c
uint32_t aster_reactor_poll(
    iroh_runtime_t runtime,
    aster_reactor_t reactor,
    aster_reactor_call_t *out_calls,
    uint32_t max_calls,
    uint32_t timeout_ms);
```

Drains up to `max_calls` from the ring into the caller-provided `out_calls` array. Returns the number of calls actually written (may be 0).

- `timeout_ms == 0` — non-blocking, return immediately whether or not calls are available.
- `timeout_ms > 0` — block up to that duration waiting for at least one call. Returns 0 on timeout.

`out_calls` must point to contiguous storage of at least `max_calls * sizeof(aster_reactor_call_t)` bytes. The buffer is filled from index 0; entries past the returned count are untouched.

**Threading:** the binding's poll thread is the only thread allowed to call this function for a given reactor. Calling from multiple threads concurrently is undefined behavior — this is the "single consumer" half of SPSC.

### `aster_reactor_submit`

```c
int32_t aster_reactor_submit(
    iroh_runtime_t runtime,
    aster_reactor_t reactor,
    uint64_t call_id,
    const uint8_t *response_ptr, uint32_t response_len,
    const uint8_t *trailer_ptr,  uint32_t trailer_len);
```

Sends the response and trailer frames for a call previously delivered by `poll`. The reactor writes both frames to the QUIC stream and finishes it.

Returns `IROH_STATUS_OK` on success or `IROH_STATUS_NOT_FOUND` if the `call_id` is unknown (already submitted, never seen, or destroyed). After this returns, the `call_id` is no longer valid.

Either pointer can be null with length 0 if you don't need that frame (rare — typically the trailer carries the RPC status).

**Threading:** safe to call from any thread, including a thread other than the poll thread.

### `aster_reactor_buffer_release`

```c
int32_t aster_reactor_buffer_release(
    iroh_runtime_t runtime,
    aster_reactor_t reactor,
    uint64_t buffer);
```

Releases a buffer obtained from a `aster_reactor_call_t` field. Each call descriptor carries three buffer IDs (`header_buffer`, `request_buffer`, `peer_buffer`) — each must be released exactly once after the binding is done with the corresponding bytes.

Returns `IROH_STATUS_OK` if the buffer was found and released, `IROH_STATUS_NOT_FOUND` otherwise.

### `aster_reactor_destroy`

```c
int32_t aster_reactor_destroy(
    iroh_runtime_t runtime,
    aster_reactor_t reactor);
```

Stops the reactor's pump task and frees its state. After this returns, no further calls can be polled or submitted on that handle. Idempotent if you wrap it carefully on the binding side.

## The call descriptor

```c
typedef struct aster_reactor_call_t {
    uint64_t call_id;          // for submit correlation
    const uint8_t *header_ptr; // owned by reactor's buffer registry
    uint32_t header_len;
    uint8_t header_flags;
    const uint8_t *request_ptr;
    uint32_t request_len;
    uint8_t request_flags;
    const uint8_t *peer_ptr;   // peer ID UTF-8 bytes (not null-terminated)
    uint32_t peer_len;
    uint8_t is_session_call;   // 1 if this is a session-mode call
    uint64_t header_buffer;    // release after reading header_ptr
    uint64_t request_buffer;   // release after reading request_ptr
    uint64_t peer_buffer;      // release after reading peer_ptr
} aster_reactor_call_t;
```

Total size: **80 bytes**. The struct is `repr(C)` with 4-byte padding after each `uint8_t` field to maintain 8-byte alignment for the following pointer/long. If your language's struct mapping computes a different size, your layout is wrong.

The three pointers (`header_ptr`, `request_ptr`, `peer_ptr`) point into a per-reactor buffer registry. They remain valid until you call `aster_reactor_buffer_release` for the corresponding buffer ID. **Holding pointers past the release call is undefined behavior.**

---

## Buffer lifetime contract

This is the most subtle part of the API. Get it wrong and you get use-after-free.

For each call delivered by `poll`:

1. The binding owns three buffer IDs (`header_buffer`, `request_buffer`, `peer_buffer`).
2. The binding may read from the corresponding pointers freely while it owns the buffers.
3. The binding **must** call `aster_reactor_buffer_release` for each buffer exactly once.
4. After release, the pointers are invalid. The binding must not retain them.

A well-behaved binding releases buffers in a `try {} finally {}` block (or your language's equivalent) so a panicking handler doesn't leak. The Java reference implementation (`bindings/java/src/main/java/com/aster/ffi/Reactor.java`) shows the pattern.

If you need to keep header bytes alive past the release (e.g., to extract a callId for trace propagation), copy them out into language-managed memory before releasing.

## Threading model

| Function | Thread |
|---|---|
| `aster_reactor_create` | any |
| `aster_reactor_poll` | exactly one dedicated thread per reactor |
| `aster_reactor_submit` | any (typically a worker pool or async task) |
| `aster_reactor_buffer_release` | any |
| `aster_reactor_destroy` | any (but stop polling first) |

The single-threaded `poll` constraint is what makes the SPSC ring lock-free. If you need to dispatch handlers concurrently, hand calls off to a worker pool from the poll thread — don't try to share the poll thread.

## Reference implementation — Java FFM

`bindings/java/src/main/java/com/aster/ffi/Reactor.java` is the reference. Key points:

- Uses Java 25 `java.lang.foreign` (FFM), not JNI.
- Static `MethodHandle` instances for the 5 functions, resolved once via `IrohLibrary.getInstance().getHandle(...)`.
- Static `CALL_LAYOUT` matching the 80-byte struct, with named offsets exposed as constants (`OFFSET_CALL_ID`, `OFFSET_HEADER_BUFFER`, etc.) for the poll loop to use.
- The `Reactor` class is `AutoCloseable` and `close()` is idempotent.
- `bufferRelease` swallows `NOT_FOUND` so double-release is benign.

A typical poll loop on the Java side looks like this:

```java
Reactor reactor = new Reactor(runtimeHandle, nodeHandle, 256);
try (Arena arena = Arena.ofConfined()) {
    MemorySegment calls = arena.allocate(Reactor.CALL_LAYOUT, 32);
    while (running) {
        int n = reactor.poll(calls, 32, 100);
        for (int i = 0; i < n; i++) {
            MemorySegment c = calls.asSlice(
                i * Reactor.CALL_LAYOUT.byteSize(),
                Reactor.CALL_LAYOUT);
            long callId = c.get(ValueLayout.JAVA_LONG, Reactor.OFFSET_CALL_ID);
            long hdrBuf = c.get(ValueLayout.JAVA_LONG, Reactor.OFFSET_HEADER_BUFFER);
            long reqBuf = c.get(ValueLayout.JAVA_LONG, Reactor.OFFSET_REQUEST_BUFFER);
            long peerBuf = c.get(ValueLayout.JAVA_LONG, Reactor.OFFSET_PEER_BUFFER);
            try {
                // ... read header_ptr/header_len, decode, dispatch, build response ...
                reactor.submit(callId, responseFrame, trailerFrame);
            } finally {
                reactor.bufferRelease(hdrBuf);
                reactor.bufferRelease(reqBuf);
                reactor.bufferRelease(peerBuf);
            }
        }
    }
} finally {
    reactor.close();
}
```

`bindings/java/src/test/java/com/aster/ReactorContractTest.java` validates the struct layout against the live native library. If you're porting to .NET or Go, write an equivalent contract test first — it catches alignment bugs before they cause use-after-free in production.

## Building a new binding

1. **Read this doc, [FFI_PLAN.md](../../ffi_spec/FFI_PLAN.md), and [new-language-binding-guide.md](./new-language-binding-guide.md).**
2. **Load the dylib.** `target/release/libaster_transport_ffi.dylib` on macOS, `.so` on Linux, `.dll` on Windows.
3. **Bind the 5 reactor functions** plus enough of the rest of the C ABI to create a runtime and node (`iroh_runtime_new`, `iroh_node_memory_with_alpns`, `iroh_poll_events` to wait for `NODE_CREATED`).
4. **Define the call descriptor struct** with explicit layout. Verify size == 80 bytes in a contract test.
5. **Write a contract test** modelled on `ReactorContractTest.java`: create a reactor, poll empty (returns 0), destroy. This validates the FFM/P/Invoke/cgo binding works against a real dylib.
6. **Wire up the dispatch loop** — read frames, decode StreamHeader (use the Aster IDL types), invoke user handlers, submit responses, release buffers.
7. **Add an integration test** that connects two nodes and round-trips a call through the reactor.

The hard parts (ring buffer correctness, pump task, buffer lifetime tracking) are already done in Rust. Your binding only needs to be a thin, type-safe wrapper.
