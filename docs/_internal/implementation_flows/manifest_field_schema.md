# Manifest Field Schema (v1)

How service contract manifests describe field types in a cross-platform
way. The schema is designed so a single manifest can drive client
codegen for Python, TypeScript, Kotlin, Java, Go, and .NET.

**Version:** `"v": 1` on the manifest JSON.

## The problem

The original manifest used Python type strings (`"type": "list[LogEntry]"`)
which other languages can't parse without a Python-specific regex parser.
Kotlin requires defaults on all dataclass fields, Go has zero-value
semantics, Java needs explicit generic types. A structured schema solves
all of these.

## Field schema

Each field in `methods[].fields` and `methods[].response_fields` is:

```json
{
  "name": "entries",
  "kind": "list",
  "item_kind": "ref",
  "item_ref": "LogEntry",
  "item_wire_tag": "mission_control/LogEntry",
  "item_nullable": false,
  "nullable": false,
  "required": false,
  "default_value": null,
  "default_kind": "empty_list",
  "properties": {}
}
```

### `kind` values

| kind | Python | Kotlin | Go | Java | TS | .NET |
|------|--------|--------|----|------|----|------|
| `string` | `str` | `String` | `string` | `String` | `string` | `string` |
| `int` | `int` | `Long` | `int64` | `long` | `number` | `long` |
| `float` | `float` | `Double` | `float64` | `double` | `number` | `double` |
| `bool` | `bool` | `Boolean` | `bool` | `boolean` | `boolean` | `bool` |
| `bytes` | `bytes` | `ByteArray` | `[]byte` | `byte[]` | `Uint8Array` | `byte[]` |
| `list` | `list[X]` | `List<X>` | `[]X` | `List<X>` | `X[]` | `List<X>` |
| `map` | `dict[K,V]` | `Map<K,V>` | `map[K]V` | `Map<K,V>` | `Record<K,V>` | `Dictionary<K,V>` |
| `ref` | `MyType` | `MyType` | `MyType` | `MyType` | `MyType` | `MyType` |
| `enum` | `str` | `MyEnum` | `string` | `MyEnum` | `string` | `MyEnum` |

### Nullable matrix

| Scenario | `nullable` | `item_nullable` | Kotlin |
|----------|-----------|-----------------|--------|
| `list[Entry]` | false | false | `List<Entry>` |
| `list[Entry]?` | true | false | `List<Entry>?` |
| `list[Entry?>` | false | true | `List<Entry?>` |
| `list[Entry?>?` | true | true | `List<Entry?>?` |

Same pattern for maps: `value_nullable` for `Map<String, Entry?>`.

### `default_kind` values

| default_kind | Python | Kotlin | Go | Java |
|-------------|--------|--------|----|------|
| `"value"` | `= <default_value>` | `= <default_value>` | (zero-value) | `.withDefault()` |
| `"empty_list"` | `field(default_factory=list)` | `= emptyList()` | `nil` | `= new ArrayList<>()` |
| `"empty_map"` | `field(default_factory=dict)` | `= emptyMap()` | `nil` | `= new HashMap<>()` |
| `"null"` | `= None` | `= null` | `nil` | `= null` |
| `"none"` | (required) | (required) | (required) | (required) |

### Type-specific keys

Only present when applicable:

| Key | When | Description |
|-----|------|-------------|
| `ref_name` | kind=ref/enum | Type name (e.g. "LogEntry") |
| `wire_tag` | kind=ref | Fory wire type tag (e.g. "mc/LogEntry") |
| `enum_values` | kind=enum | List of possible values (e.g. ["online","offline"]) |
| `item_kind` | kind=list | Element kind |
| `item_ref` | kind=list, item_kind=ref | Element type name |
| `item_wire_tag` | kind=list, item_kind=ref | Element wire tag |
| `item_nullable` | kind=list | Can elements be null? |
| `key_kind` | kind=map | Key kind (usually "string") |
| `value_kind` | kind=map | Value kind |
| `value_ref` | kind=map, value_kind=ref | Value type name |
| `value_nullable` | kind=map | Can values be null? |
| `properties` | always | Extensible dict for future use |

## Producing the schema (server side)

**Python:** `build_field_v1()` in `contract/manifest.py`. Introspects
dataclass fields using `dataclasses.fields()` and `get_type_hints()`.
Maps Python types to kind values via `_classify_type()`.

**TypeScript:** `extractFields()` in `high-level.ts`. Infers types from
runtime `typeof` checks on instantiated default values. Less rich than
Python (can't detect enum values or ref wire tags from runtime alone).

## Consuming the schema (codegen side)

**Python codegen:** `_py_type_from_field()` and `_py_default_from_field()`
in `cli/aster_cli/codegen.py`. Reads `kind`, `ref_name`, `item_kind` etc.
and produces Python type annotations and default expressions.

**For other languages:** Read the same JSON manifest, switch on `kind`,
emit language-idiomatic types and defaults. The schema contains enough
information for any statically-typed language.

## Legacy compatibility

Manifests without `"v"` are auto-upgraded on read. The `from_json()`
method detects fields with `"type"` but no `"kind"` and runs
`upgrade_legacy_field()` to convert them.

## Wire compatibility note

The field schema is NOT part of the wire protocol. It lives in the
manifest JSON (stored in the registry or generated by `aster contract gen`).
Changing the schema does not affect contract identity hashes.

## Implementation checklist for new bindings

- [ ] Manifest producer: introspect service types, emit v1 field schema
- [ ] Manifest consumer: read v1 fields, emit language-idiomatic types
- [ ] Handle all `kind` values (string/int/float/bool/bytes/list/map/ref/enum)
- [ ] Handle `nullable` and `item_nullable`/`value_nullable`
- [ ] Handle all `default_kind` values (value/empty_list/empty_map/null/none)
- [ ] Handle `enum_values` for enum codegen
- [ ] Handle legacy manifests (auto-upgrade on read)
- [ ] Set `"v": 1` on produced manifests
